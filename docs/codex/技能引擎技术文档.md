# Kayou 技能引擎技术文档

> 基于 2026-04-10 仓库代码与现有需求文档整理。本文只讨论“技能系统”的技术底座、规则结构与后续开发约束，不包含具体代码实现。

## 1. 文档目标

Kayou 当前已经具备以下内容生产链路：

1. 人物管理
2. 人物背景/外形/绘图提示词的大模型生成
3. AI 绘图、裁切、WebP 素材沉淀
4. 卡牌属性编辑
5. 卡牌模板制作
6. 24 个小游戏的参数公式配置与试玩

但“技能”目前仍停留在：

1. `card_attributes` 表里最多 3 个技能槽位
2. 大模型只生成技能名字与技能描述
3. 还没有统一技能模板、技能分类、技能触发时机、技能效果原子、技能数值公式、技能运行时与发布治理规则层

所以这份文档的目标不是直接定义某一个技能，而是先把未来技能系统需要的“底座”尽量列全，方便后续逐步落地。

## 2. 项目现状与边界

### 2.1 已有事实

| 维度 | 当前现状 | 对技能引擎的影响 |
| --- | --- | --- |
| 角色主数据 | `characters` 表保存人物、背景、外形、图像路径、WebP 路径 | 技能需要能引用角色内容、题材、形象、系列信息 |
| 卡牌属性 | `card_attributes` 表已有 `series_name`、`faction_name`、`rarity`、四维属性、元素、技能1/2/3 名字与描述 | 技能引擎第一阶段必须兼容现有表结构，不能强依赖全新战斗系统 |
| 大模型链路 | 当前已能批量生成阵营、稀有度、四维属性、元素、技能文案 | 技能引擎要支持“文案先生成，再结构化归类” |
| 卡牌制作 | 当前已有全局卡牌模板与字段显隐/布局/字体/特效配置 | 技能引擎要能为卡牌成品输出稳定的技能展示信息 |
| 小游戏系统 | `games` 表存 `formula_config`，前端已有 24 个小游戏组件，统一使用敌我四维属性做公式计算 | 技能引擎必须与小游戏保持清晰边界：小游戏只读取四维属性，不直接读取技能 |
| 配置存储风格 | 当前大量配置存于 `settings` 表 | 技能引擎可以复用配置驱动思路，但正式技能数据不建议长期只塞 `settings` |
| 运行状态 | 还没有对战、回合、血量、卡组、战场等完整战斗域模型 | 技能引擎第一期不应假设完整 TCG/SLG 战斗规则已经存在 |

### 2.2 这套技能引擎在 Kayou 中的定位

Kayou 的技能引擎不应被设计成“只服务未来战斗结算”的孤立模块，而应该是一个统一规则层，连接以下四块：

1. 内容生产层：把大模型生成的技能描述沉淀成结构化技能资产。
2. 数值规则层：把技能从纯文本描述转成可计算、可校验、可复用的规则对象。
3. 展示与运营层：支持后台编辑、模板复用、版本管理、平衡调整、审核发布。
4. 边界约束层：明确技能系统与小游戏系统、卡牌属性系统、未来战斗系统之间的数据边界。

### 2.3 小游戏边界

结合当前仓库和你的补充约束，Kayou 里必须明确以下边界：

1. 小游戏只读取四维属性：`force_value`、`intellect_value`、`speed_value`、`stamina_value`
2. 技能不直接绑定 `game_key`
3. 技能不直接修改小游戏公式参数
4. 技能不直接修改小游戏得分、评级或结算结果
5. 技能系统与小游戏系统当前是解耦关系，唯一公共桥梁是四维属性

这意味着后续做技能引擎时，不能把“技能直接作用小游戏”写进一线规则设计里，否则会和当前项目边界冲突。

### 2.4 当前已知技能数量约束

结合当前需求文档与现有后端逻辑，Kayou 已经隐含了如下规则：

1. `N / R / CP / SR / PR` 默认 1 个技能
2. `SSR / HR / UR` 默认 2 个技能
3. 当前后端在 AI 生成属性时，实际上只会稳定生成前 1-2 个技能
4. `skill3` 更适合作为未来扩展槽位，而不是第一期强行启用

这意味着技能引擎第一期应该优先做“1-2 个核心技能稳定结构化”，不要一开始就把三槽甚至更多槽位全部做重。

### 2.5 第一原则

第一期技能系统最重要的不是“效果做得多炫”，而是先解决三件事：

1. 技能可结构化
2. 技能可复用
3. 技能可校验

## 3. 设计原则

1. 配置优先：尽量把技能设计成数据驱动，而不是写死在单个页面或单个功能模块里。
2. 兼容当前数据：必须兼容现有 `skill1/skill2/skill3` 三槽展示方式。
3. 先描述归一，再运行落地：先建立技能模板库和规则库，再逐步接入运行时。
4. 小步扩展：先支持结构化模板、条件判断、基础效果与版本管理，再扩展复杂连锁。
5. 规则可解释：每个技能都要能回答“什么时候触发、对谁生效、怎么计算、持续多久、如何叠加”。
6. 防止失控：所有数值、概率、时长、次数、乘区都要有钳制和上限。
7. 内容与表现分离：技能规则、技能文案、技能展示皮肤、卡面排版应分层。
8. 适配 LM 工作流：允许先生成自然语言，再由规则层映射为模板和参数。
9. 小游戏边界清晰：小游戏只读取四维属性，技能不直接进入小游戏公式与结算链路。

## 4. 技能底座总览

下面按“尽量穷举”的方式罗列技能引擎需要的非代码底座。优先级说明如下：

- `P0`：建议现在就定规范，否则后面越做越乱
- `P1`：第一期技能系统就会用到
- `P2`：第二期或接入正式玩法时需要
- `P3`：高级能力，可后补

### 4.1 内容标识与资产底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 技能唯一标识 | 为每个技能定义稳定 `skill_id`，不能只靠技能名识别 | 当前技能名来自大模型，重名风险高 | P0 |
| 技能版本号 | 同一技能支持多版本迭代 | 后期平衡调整不能覆盖历史配置 | P0 |
| 技能来源标记 | 标记技能来自 LM 生成、人工录入、模板复制、导入文件 | 方便运营审计与回溯 | P0 |
| 技能状态 | 草稿、待审核、已发布、已停用、已归档 | 后台运营需要内容生命周期 | P0 |
| 展示文案层 | 名称、短描述、长描述、提示文案、战报文案分离 | 卡牌展示、后台详情、日志展示需求不同 | P1 |
| 技能标签体系 | 题材标签、系列标签、阵营标签、元素标签、玩法标签 | 方便检索、筛选、模板复用 | P1 |

| 技能槽位语义 | 主技能、被动技能、觉醒技能、队长技等 | 当前只有 skill1/2/3，后期需要语义化 | P1 |
| 技能表现资源索引 | 图标、边框、关键词、特效建议、颜色方案 | 方便后续卡牌制作与前端展示统一 | P2 |
| 多语言/别名层 | 中文名、内部名、搜索别名、拼音检索名 | 便于后台搜索和未来扩展 | P3 |

### 4.7 效果原子底座

技能不应直接写成一大段不可拆的逻辑，而应由“效果原子”组合。下面是建议尽量预留的效果原子全集。

| 效果原子 | 说明 | 与 Kayou 的关系 | 优先级 |
| --- | --- | --- | --- |
| 属性加成 | 提高或降低武力/智力/速度/体力 | 最基础，能直接兼容现有四维字段 | P0 |
| 属性倍率加成 | 对基础属性做乘法修正 | 适合高级稀有度技能 | P1 |
| 元素改写 | 修改元素、追加元素标签、元素临时附着 | 便于做题材联动 | P2 |
| 阵营/系列加成 | 同阵营或同系列对象获得收益 | 内容题材常见 | P2 |
| 参数修正 | 直接修改技能自身参数、触发阈值、持续回合、范围等 | 适合作为模板通用能力 | P0 |
| 公式修正 | 在原技能公式基础上追加项、倍率、钳制 | 适合复杂模板扩展 | P1 |
| 结算修正 | 调整伤害结算、奖励结算或胜负结算 | 通用结算型能力需要 | P1 |
| 次数修正 | 增加触发次数、使用次数、连锁次数或容错次数 | 很多技能都需要频控 | P1 |
| 时间修正 | 调整持续时间、冷却时间、准备时间 | 常见通用能力 | P1 |
| 概率修正 | 提高暴击率、命中率、掉落率、触发率 | 后期扩展通用 | P2 |
| 护盾/减伤 | 适合未来战斗域 | 当前可先不落代码，但底座要预留 | P2 |
| 回复/续航 | 治疗、恢复、清除负面状态 | 战斗域预留 | P2 |
| 控制/限制 | 沉默、冻结、禁疗、禁触发、禁加成 | 复杂玩法预留 | P3 |
| 反制/免疫 | 免疫负面、反弹效果、打断触发 | 战斗域或竞赛域会需要 | P3 |
| 状态附加 | 给目标挂 Buff、Debuff、标记、层数 | 运行时扩展的基础 | P2 |
| 复制/偷取 | 复制参数、偷取加成、镜像技能 | 高复杂度技能预留 | P3 |
| 转化/兑换 | 把某属性按比例转成另一个属性、护盾值或状态值 | 适合表达成长与资源交换 | P2 |
| 条件解锁 | 达成条件后激活第二段技能 | 觉醒/进阶玩法需要 | P2 |
| 再触发 | 额外执行一次某效果或追加一次判定 | 连锁玩法常见 | P2 |
| 冷却控制 | 重置冷却、缩短冷却、增加可用次数 | 技能节奏控制基础 | P2 |
| 日志输出 | 为结算日志写入标准说明片段 | 便于调试和用户理解 | P1 |

### 4.8 数值与公式底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 统一变量命名空间 | 建议与现有四维字段保持同名映射，如 `force_value`、`intellect_value`、`speed_value`、`stamina_value`，避免与小游戏公式变量混用 | 保持规则边界清晰 | P0 |
| 参数上下限钳制 | 概率、时长、次数、倍率都要有上下限 | 防止技能数值失控 | P0 |
| 数值单位规范 | 秒、次、倍率、比例、像素、格/秒、分数 | 便于编辑器提示与校验 | P1 |
| 加法/乘法乘区规范 | 先加后乘还是先乘后加要固定 | 没规范会严重影响平衡 | P1 |
| 四舍五入规范 | 向上取整、向下取整、保留几位小数 | 技能计算与展示都要统一 | P1 |
| 默认值与回退值 | 技能公式失败时回退默认值 | 运行时必须稳定兜底 | P0 |
| 公式校验规则 | 非法变量、除零风险、空公式、注入风险、死循环风险 | 后台可编辑公式必须有防护 | P1 |
| 公式片段库 | 常用计算片段抽成片段，如“敌我差值倍率” | 便于后续快速拼装 | P2 |
| 强度预算库 | 不同模板的推荐值区间、成长系数、概率系数 | 方便大模型和人工设计共用 | P1 |
| 随机种子策略 | 是否需要可复现随机、测试随机、正式随机 | 调试与回放需要 | P2 |
| 数值快照机制 | 记录技能触发前后属性与状态变化 | 方便平衡与排错 | P1 |

### 4.9 状态与持续时间底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 即时效果 | 触发即生效，立即结算 | 第一阶段最容易落地 | P0 |
| 持续效果 | 持续若干回合、若干局、若干阶段 | 后期玩法扩展需要 | P2 |
| 场景内效果 | 仅当前对局或当前阶段有效 | 便于控制技能生命周期 | P1 |
| 场景外效果 | 对整局、整轮、整场活动生效 | 未来活动规则可用 | P2 |
| 层数系统 | Buff 层数、标记层数、叠层上限 | 常见技能基础 | P2 |
| 唯一性规则 | 同名不可叠、同源刷新、异源叠加 | 解决冲突 | P2 |
| 覆盖规则 | 新效果覆盖旧效果或合并旧效果 | 编辑与运行都要统一 | P2 |
| 失效条件 | 结束时机、触发次数耗尽、被驱散、离场、对局结束 | 保证生命周期闭环 | P2 |

### 4.10 冷却、费用与资源底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 使用次数限制 | 每局一次、每回合一次、每场对局一次 | 很多强技能都需要频控 | P1 |
| 冷却回合/冷却阶段 | 在若干事件后恢复可用 | 未来正式玩法常见 | P2 |
| 充能层 | 通过达成条件积累使用次数 | 适合成长型或蓄力型技能 | P2 |
| 费用类型 | 扣属性、扣分、扣次数、弃用某加成 | 有助于高风险高收益技能设计 | P3 |
| 资源池预留 | 怒气、能量、连击点、专注度等未来资源 | 虽然当前没有，但底座应预留 | P3 |

### 4.11 结算顺序与冲突处理底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 优先级体系 | 技能、状态、属性修正、结算修正按固定顺序跑 | 防止多技能同时生效时混乱 | P1 |
| 前置结算与后置结算 | 先改属性再算效果，还是先算效果再改结算 | 复杂技能场景尤其重要 | P1 |
| 同步与异步区分 | 编辑期计算、开局计算、局中事件计算、结算回放计算分开 | 便于稳定实现 | P1 |
| 冲突仲裁 | 同一属性或同一状态被多个技能修改时如何叠加 | 复杂技能体系一定会遇到 | P1 |
| 最大递归深度 | 防止技能套技能无限触发 | 复杂连锁必备 | P2 |
| 幂等控制 | 同一事件重复到达时不应重复发奖或重复写状态 | 后期接口联动很关键 | P2 |
| 审计日志链 | 每次结算输出“谁在什么时机改了什么值” | 调试和运营都离不开 | P1 |

### 4.12 小游戏边界与兼容底座

这一层不是让技能直接介入小游戏，而是把边界写清楚，防止后续开发把两套系统重新耦合在一起。

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 四维属性快照 | 小游戏只读取敌我四维属性快照 | 这是当前唯一合法接入面 | P0 |
| 技能隔离边界 | 技能定义不直接绑定 `game_key` | 避免职责串层 | P0 |
| 参数链路隔离 | 技能不直接修改 `formula_config` 输出参数 | 保持小游戏公式系统单一可信输入 | P0 |
| 结算链路隔离 | 技能不直接修改 0-5 分或评级 | 保持小游戏结算独立 | P0 |
| 数据同步规则 | 若未来需要间接影响，只能先落到四维属性，再由小游戏读取 | 防止出现双轨规则 | P1 |
| 平衡协同规则 | 技能系统与小游戏系统分别平衡，但要共享四维属性预算 | 避免一边改强一边失控 | P1 |
| 边界测试 | 校验小游戏运行时未读取技能字段 | 防止后续偷偷耦合 | P1 |
| 版本解耦 | 技能版本更新不应强制小游戏参数版本同步 | 降低维护成本 | P2 |

### 4.13 展示与卡面联动底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 技能展示摘要 | 短描述、关键词、数值摘要、图标文本 | 卡牌展示空间有限 | P1 |
| 技能长描述 | 用于详情页、后台编辑页、结算说明 | 展示与规则不能混成一段 | P1 |
| 技能关键词高亮 | 如“加速”“护盾”“暴击”“保底分” | 便于玩家快速理解 | P2 |
| 技能显示顺序 | 主技能优先、被动靠后、觉醒技单列 | 当前卡面模板已经支持字段排版 | P1 |
| 技能显示隐藏规则 | 某些内部字段只在后台可见 | 运营和玩家视图应分离 | P2 |
| 技能图标语义 | 用图标映射功能标签 | 后期 UI 成本更低 | P2 |

### 4.14 编辑、审核与运营底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 技能库后台 | 列表、筛选、编辑、克隆、停用、发布 | 没有后台就很难规模化维护 | P0 |
| 模板库后台 | 管理底层技能底板及其参数槽 | 当前需求已明确要做“底板”沉淀 | P0 |
| 结构化映射后台 | 把 LM 生成文案映射到模板和参数 | 这是 Kayou 的现实入口 | P0 |
| 技能审核流 | 生成后待审、驳回、修改、发布 | 控制内容质量 | P1 |
| 导入导出 | 支持 CSV/Excel/JSON 批量维护 | 当前人物与属性都已有导入导出思路 | P1 |
| 技能差异对比 | 查看某次改动前后差异 | 平衡调整常见需求 | P1 |
| 技能复用推荐 | 基于标签或模板推荐相似技能 | 方便后期批量生产 | P2 |
| 发布批次管理 | 技能按版本包发布 | 防止线上配置半更新 | P2 |
| 权限分层 | 设计可编辑、审核可发布、运营可停用 | 中后台长期必须有 | P2 |
| 审计日志 | 谁在何时改了什么 | 运维和协作必备 | P1 |

### 4.15 质量、测试与平衡底座

| 底座 | 说明 | 为什么 Kayou 需要 | 优先级 |
| --- | --- | --- | --- |
| 结构校验器 | 校验技能是否缺字段、字段冲突、模板参数缺失 | 第一阶段就该有 | P0 |
| 规则校验器 | 校验触发时机、目标、条件、效果组合是否合法 | 避免运行时报错 | P1 |
| 数值校验器 | 校验是否超上限、公式是否危险、概率是否失衡 | 技能参数最怕失控 | P1 |
| 模拟测试样例 | 录入若干标准敌我属性样例做跑批测试 | 便于看技能是否过强 | P1 |
| 回放日志 | 一次技能触发链的完整解释 | 排查问题和做平衡都需要 | P1 |
| 统计指标 | 使用率、胜率、平均触发次数、平均收益、异常日志 | 后期迭代必须依赖数据 | P2 |
| 平衡画像 | 每个模板、每个稀有度、每类玩法的强度画像 | 方便做版本平衡 | P2 |
| 线上开关 | 一键停用异常技能或回退版本 | 运营安全必备 | P2 |

## 5. 建议的核心实体模型（非 SQL）

为了避免后续技能系统继续只靠 `skill1_name`、`skill1_desc` 这种平铺字段硬扛，建议至少在概念层引入以下实体。这里不是数据库脚本，只是未来开发时应有的对象边界。

| 实体 | 作用 | 建议包含的信息 |
| --- | --- | --- |
| SkillDefinition | 技能主定义 | `skill_id`、名称、描述、标签、状态、来源、版本、模板 ID |
| SkillTemplate | 技能底板 | 模板类型、参数槽定义、默认值、适用域、适用时机、强度预算 |
| SkillEffectAtom | 技能效果原子 | 效果类型、目标、参数、优先级、钳制规则 |
| SkillTrigger | 技能触发器 | 事件名、触发次数、概率、前置条件 |
| SkillCondition | 技能条件 | 属性比较、标签命中、状态命中、组合逻辑 |
| SkillFormula | 技能公式 | 输入变量、公式文本、默认值、单位、上下限 |
| SkillBinding | 技能装配关系 | 哪张卡/哪个角色/哪个槽位装了哪个技能 |
| SkillVersion | 技能版本快照 | 版本号、生效时间、改动说明、回滚点 |
| SkillPublishRecord | 发布记录 | 发布批次、发布人、发布时间、影响范围 |
| SkillTestCase | 测试用例 | 输入条件、期望输出、属性快照、结算快照 |
| SkillTelemetry | 统计指标 | 触发次数、平均收益、问题日志 |

## 7. 与当前仓库最匹配的落地顺序

### 7.1 第一阶段：先把“技能描述”收口成“技能模板”

建议目标：

1. 先不做完整战斗引擎
2. 先把大模型生成的技能名字与描述，映射成少量标准模板
3. 每个技能模板只允许填固定参数
4. 技能先只作用于卡牌自身规则数据、四维属性衍生逻辑与展示结构，不接小游戏运行时

这一阶段的最小闭环应包含：

1. 技能唯一 ID
2. 技能模板库
3. 模板参数槽
4. 技能后台管理
5. 技能结构校验
6. 模板匹配与人工审核

### 7.2 第二阶段：接入正式技能运行时

建议目标：

1. 技能可在统一事件上触发
2. 技能可对属性、状态、结算产生标准化影响
3. 技能可输出触发日志与测试回放

这时最重要的是：

1. 标准事件字典
2. 效果修改优先级
3. 钳制规则
4. 触发日志

### 7.3 第三阶段：扩展到正式玩法或战斗域

等 Kayou 后续进入真正“可玩的网页游戏”阶段，再逐步补：

1. 回合制事件
2. 伤害与治疗
3. Buff / Debuff
4. 冷却与资源
5. 连锁与反制
6. 对战日志与回放

## 8. 明确不建议第一期就做的内容

下面这些能力虽然未来可能会用到，但不建议现在就全量落地：

1. 完整战场对象模型
2. 复杂召唤物系统
3. 多资源池战斗系统
4. 技能脚本自由编程
5. 过度开放的表达式引擎
6. 无上限的技能连锁机制

原因很简单：Kayou 当前最成熟的是“内容生产链”和“属性配置链”，而小游戏参数链应继续保持独立，所以技能系统应该先服务结构化、模板化与后续正式玩法，而不是先把边界打乱。

## 9. 总结

对 Kayou 来说，技能引擎最核心的底座不是某一个技能公式，而是下面这五件事：

1. 技能要有稳定标识和版本
2. 技能要能从自然语言描述沉淀成模板
3. 技能要围绕当前已有的四维属性、稀有度、元素、阵营与卡牌展示字段展开，并与小游戏保持清晰边界
4. 技能要有统一的触发、目标、条件、效果、结算规范
5. 技能要能被后台编辑、审核、测试、发布、回滚

如果按这个方向推进，后续无论是继续做卡牌属性、沉淀技能模板，还是最终进入网页游戏玩法阶段，技能系统都不会推倒重来，只会不断往上加能力。
