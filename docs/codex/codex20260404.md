# Kayou 项目文档

本文件基于 2026-04-04 对仓库代码的静态检查整理而成，目标是帮助后续开发、维护和交接时快速理解项目现状。

## 1. 项目定位

`kayou` 当前是一个前后端分离的管理后台项目，核心目标不是直接提供游戏玩法，而是先完成“卡牌图片 AI 生成工作流”的管理端能力。按照现有需求，当前阶段重点是：

- 管理员登录
- 人物信息维护
- 通过大模型生成背景描述和外形描述
- 通过大模型生成人物绘图提示词
- 调用外部绘图服务生成图片
- 对图片进行裁切、筛选和删除
- 将选中的图片转换为 WebP 素材

项目中还预留了“卡牌属性编辑”入口，但目前只是占位页面，尚未进入真实开发。

## 4. 当前功能概览

### 4.1 登录与鉴权

- 后端提供 `POST /api/auth/login`
- 登录成功后返回 JWT，过期时间为 `24h`
- 前端将 token 保存到 `localStorage`
- 除登录接口外，人物、LLM、绘图、图片处理相关接口全部要求 `Authorization: Bearer <token>`

当前实现中，管理员账号密码写死在后端：

- 用户名：`admin`
- 密码：`kayouadmin`

### 4.2 人物管理

人物管理对应前端 `CharacterTable.vue` 与后端 `routes/characters.js`，目前支持：

- 刷新列表
- 新增单个人物
- 编辑人物字段
- 删除单条记录
- 批量删除
- CSV 导入
- CSV 导出
- 展示背景、外形、提示词、图片数量、WebP 数量

新增人物时：

- 只要求输入 `name`
- 后端自动调用拼音服务生成 `pinyin`

CSV 导入时支持的列名包括：

- `name`
- `人物名字`
- `名字`
- `pinyin`
- `人物名字拼音`

### 4.3 大模型文本工作流

LLM 功能分为两部分：

1. 配置模型接口
2. 调用模型生成文本

配置项保存到数据库 `settings` 表中，包括：

- `llm_base_url`
- `llm_api_key`
- `llm_model`

后端假设上游接口兼容 OpenAI 风格的：

- `POST {base_url}/chat/completions`

文本生成分成两类：

- 人物背景介绍 + 外形描述
- 绘图提示词（`nb2_prompt` 与 `mj_prompt`）

生成方式为逐条处理，并通过 `text/event-stream` 持续向前端推送进度。

### 4.4 AI 绘图工作流

绘图逻辑位于 `backend/routes/drawing.js`，主要流程如下：

1. 从数据库读取 MX AI 配置
2. 读取人物的 `mj_prompt`
3. 调用 MX AI 提交绘图任务
4. 轮询任务状态直到成功或失败
5. 下载返回图片链接
6. 按人物拼音创建目录并保存文件
7. 将图片路径写回 `characters.image_paths`

MX 配置保存在 `settings` 表：

- `mx_api_key`
- `mx_base_url`

生成后的原始图片落在：

- `backend/assets/image/<人物拼音>/`

数据库中保存的路径格式为：

- `/assets/image/<人物拼音>/<文件名>`

### 4.5 图片裁切

图片裁切通过 `POST /api/images/crop` 完成，支持两种模式：

- 自动裁切：以中心区域裁出 `2:3` 比例
- 手动裁切：前端输入 `left/top/width/height`

当前实现会直接覆盖原始图片文件，不会额外保存裁切前版本。

### 4.6 WebP 转换

WebP 转换由 `backend/routes/images.js` 提供，支持：

- 选中部分图片转换
- 将某人物全部图片转换为 WebP
- 删除已生成的 WebP

WebP 文件统一保存在：

- `backend/assets/webp/`

命名策略：

- 单张通常为 `<pinyin>.webp`
- 多张时追加 `-1`、`-2` 等后缀

数据库中保存的路径格式为：

- `/assets/webp/<文件名>`

### 4.7 卡牌属性编辑

前端已预留 `/admin/attributes` 页面，但当前只展示一个禁用按钮：

- `卡牌属性编辑（开发中）`

目前没有相关数据库结构、接口或页面表单实现。

## 5. 前端结构说明

### 5.1 页面路由

前端路由很简单：

- `/login`：登录页
- `/admin`：后台布局页
- `/admin/workflow`：AI 工作流主页面
- `/admin/attributes`：属性编辑占位页

未匹配路径会重定向到 `/admin`。

### 5.2 状态管理

Pinia 中当前只有一个 `auth` store，负责：

- 读取本地 token
- 设置 token
- 清除 token

没有引入更复杂的全局业务状态，列表、进度日志、弹窗状态都在各自组件内部维护。

### 5.3 接口层

接口请求主要分两种：

- 基于 Axios 的普通请求
- 基于 `fetch` 的流式请求

Axios 实例特点：

- 固定 `baseURL = http://localhost:3174/api`
- 自动附加 Bearer token
- 遇到 `401` 时自动清空登录状态并跳回登录页

流式接口没有封装到统一 API 层，而是直接在组件中用 `fetch()` 读取 SSE 数据流。

### 5.4 关键组件

#### `CharacterTable.vue`

负责：

- 人物列表展示
- 新增、编辑、删除、批量删除
- CSV 导入导出
- 触发“生成描述”
- 触发“生成提示词”
- 展示流式生成进度

#### `LLMConfigPanel.vue`

负责：

- 保存 LLM 配置
- 测试 LLM 连接
- 保存 MX AI 配置

#### `AIDrawingPanel.vue`

负责：

- 批量绘图
- 单个人物重绘
- 展示原始图片缩略图
- 预览大图
- 删除原图
- 自动裁切与手动裁切

#### `WebpConvertPanel.vue`

负责：

- 选择原图
- 转换选中图片
- 全部转换
- 预览 WebP
- 删除 WebP

数据库为 SQLite，使用 `better-sqlite3`，特点如下：

- 启用 `WAL` 模式
- schema 初始化写在 `backend/db/init.js`

当前只有两张业务表。

## 9. 核心业务流程

### 9.1 从人物名字到可用素材

1. 管理员登录后台。
2. 新增人物或导入 CSV。
3. 后端为人物自动生成拼音。
4. 配置大模型接口。
5. 批量生成人物背景介绍和外形描述。
6. 批量生成 `nb2_prompt` / `mj_prompt`。
7. 配置 MX AI。
8. 批量生成原始图片并保存到 `/assets/image/`。
9. 对不满意的原图进行删除、重绘、裁切。
10. 把满意的原图转换为 WebP 并保存到 `/assets/webp/`。

### 9.2 流式进度反馈

项目里“生成描述”“生成提示词”“绘图”这三类耗时任务都采用同一种前端交互模式：

- 前端通过 `fetch` 发起请求
- 后端返回 `text/event-stream`
- 前端逐行解析 `data: ...`
- 将每条处理结果追加到本地进度日志
- 收到 `[DONE]` 后结束

这种实现方式简单直接，适合当前单机后台场景。

## 12. 已知问题与风险点

以下内容不是“必须立刻修改”的结论，而是当前实现中值得后续优先关注的地方。

### 12.1 导出接口前端大概率不可用

前端导出使用的是：

- `window.open('http://localhost:3174/api/characters/export')`

但后端的 `/api/characters/export` 被鉴权中间件保护，需要 Bearer token。`window.open` 这类调用无法自动附带当前登录 token，因此在真实浏览器环境里很可能直接返回 `401`。

### 12.2 前端把后端地址写死为 `http://localhost:3174`

以下场景都直接写死了主机与端口：

- Axios `baseURL`
- SSE `fetch`
- 预览图片地址
- 导出地址

这会导致：

- 换端口时需要改代码
- 部署到非本机环境时需要重新适配
- 前后端反向代理接入不方便

### 12.3 登录凭据与 JWT Secret 写死在代码里

当前实现为了快速起步，把以下信息直接写在代码中：

- 管理员用户名
- 管理员密码
- JWT Secret

这在开发阶段可以接受，但继续推进到多人协作、测试环境或线上环境时会带来明显安全风险。

### 12.4 “修改名字”不会自动重算拼音

后端在新增人物时会自动生成拼音，但编辑人物时如果只改了 `name` 而没有同步更新 `pinyin`，数据库中可能保留旧拼音。由于拼音还承担目录命名职责，这会造成名字与文件目录不一致。

### 12.5 图片裁切会直接覆盖源文件

当前裁切逻辑没有保留原始备份，因此：

- 误裁切后无法回退
- 二次裁切基于的是上一次裁切结果

### 12.6 目前没有测试体系

仓库中尚未看到：

- 单元测试
- 接口测试
- 端到端测试
- lint / format / CI 流程

因此当前代码更适合快速验证需求，不适合直接依赖自动化质量保障。

### 12.7 前端样式文件存在模板遗留代码

`frontend/src/style.css` 仍保留了大量 Vite 模板默认样式，但当前页面主要依赖 Element Plus 和组件内联样式。该文件不会影响主功能理解，但说明前端基础清理尚未完成。

## 13. 后续开发建议

如果后续继续开发，建议优先按下面顺序推进：

1. 修复导出鉴权问题，并统一前端 API 地址配置方式。
2. 为后端补充 `start` / `dev` 脚本，整理启动说明。
3. 把登录凭据、JWT Secret、外部 API 配置迁移到环境变量。
4. 补齐“卡牌属性编辑”对应的数据结构、接口和页面。
5. 为关键流程补最基础的接口测试与手工回归清单。
6. 重新梳理图片与 WebP 的命名、覆盖、删除和回滚策略。

## 14. 总结

从代码结构来看，这个项目已经完成了一个比较明确的第一版管理后台原型，尤其是“人物维护 -> 文本生成 -> 提示词生成 -> AI 绘图 -> 图片裁切 -> WebP 转换”这条链路已经闭环。当前最需要做的不是大规模重构，而是围绕配置、安全性、可部署性和未完成模块，继续把现有原型打磨成更稳定的内部工具。
